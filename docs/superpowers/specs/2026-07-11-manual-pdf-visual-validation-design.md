# Manual del Color Vivo: Dev-Only PDF Visual Validation — Design

## Context

Phase 2c (`docs/superpowers/specs/2026-07-10-phase2c-manual-ssr-webpage-design.md`)
shipped 87 authenticated SSR pages for *Manual del Color Vivo*. A subsequent
content-authoring pass (`docs/superpowers/specs/2026-07-10-manual-content-authoring-design.md`)
has since transcribed the book's prose into most page bodies.

This spec covers a small developer tool to make that transcription easier to
verify (already-transcribed pages, and any future edits/corrections): a
**side-by-side comparison view, visible only in `Rails.env.development?`**,
showing the source PDF pages for a section next to the section's current
rendered content, so a developer can visually check transcribed prose against
the original book layout without alt-tabbing to a PDF viewer.

The source PDF lives at `project/*.pdf` (gitignored, 136 pages, local-only —
already referenced this way in the Phase 2c spec).

## Goals

- On every manual section page (not the `Index`/Contenido page), when running
  in development, show a two-column layout: left = the PDF page(s) for that
  section, right = the section's current rendered content (unchanged).
- A toggle button collapses the left column back to a single-column view.
- Zero footprint outside development: no new route reachable, no new prop
  sent, `ManualLayout` renders exactly as it does today in production/test.
- One-time, offline mapping from each of the 87 section paths to its PDF page
  range, generated by a rake task and committed as plain data (page numbers
  only — no PDF content is committed).

## Out of scope

- Production or test-env behavior changes — this is a development-only tool.
- Any change to the Phase 2c content model, routes, or `Manual` table of
  contents structure.
- OCR text extraction, prose diffing, or any automated correctness checking
  of transcribed content — this is a *visual* aid, not a checker.
- Handling PDF page renumbering after this feature ships (if a future edition
  of the PDF changes, the map must be regenerated by rerunning the rake
  task).

## Design

### 1. PDF → page-range map (one-time rake task)

`lib/tasks/manual.rake` gets a new task, `manual:map_pdf_pages`, run once by
a developer locally (requires the gitignored PDF to be present) and never at
app boot or request time.

Algorithm, walking `Manual.paths` in book order against PDF pages (via
`pdftotext`) in order:

1. Normalize text: strip accents, collapse whitespace, downcase, strip
   punctuation. Apply the same normalization to each title and to each PDF
   page's text.
2. **Fast path:** substring-match the normalized title against the
   normalized text of the current/next few PDF pages (scanning forward only
   — never backward — from the last matched page, since titles appear in
   book order).
3. **Fallback:** if no substring match, compute Levenshtein distance between
   the normalized title and normalized candidate lines on the scanned pages;
   accept the best candidate if distance ≤ 2. (Plain ~10-line DP
   implementation in the rake task itself — no new gem for a one-off script.)
4. If neither matches within a reasonable forward-scan window, **halt loudly**
   with the offending title and the last successfully matched page, so a
   human resolves that one entry by hand and reruns.
5. A section's page range is `[start_page, next_section's start_page - 1]`
   (last section's range ends at page 136).

Output: `config/manual_pdf_pages.yml`, e.g.:

```yaml
el-origen-del-color/introduccion: [5, 6]
el-origen-del-color/el-color-en-la-naturaleza: [7, 8]
...
```

Loaded via a new `Manual.pdf_page_range(path)` (returns `[start, end]` or
`nil`), read from this YAML once at boot. This file is committed — it's just
integers, not PDF content, so it carries no copyright/gitignore concern.

### 2. Dev-only rasterize route

New route, drawn **only** when `Rails.env.development?`:

```ruby
if Rails.env.development?
  get "/dev/manual_pdf_pages/:page.png", to: "manual_pdf_pages#show",
      constraints: { page: /\d+/ }
end
```

`ManualPdfPagesController#show`:

- Parses `params[:page]` as an integer, rejects (404) anything outside
  `1..136`.
- Checks `tmp/manual_pdf_pages/<page>.png`; if missing, shells out to
  `pdftoppm` via `Open3.capture2` with an **argv array** (never string
  interpolation into a shell):
  `["pdftoppm", "-png", "-singlefile", "-r", "150", "-f", page.to_s, "-l",
  page.to_s, pdf_path.to_s, "tmp/manual_pdf_pages/#{page}"]`. The `-singlefile`
  flag is required — without it `pdftoppm` appends its own page-number suffix
  to the output filename, which would not match the `<page>.png` name the
  controller expects to serve.
- Sends the cached PNG with `send_file`.
- Defense in depth: the controller itself also raises
  `ActionController::RoutingError` unless `Rails.env.development?`, so even
  a routes.rb mistake can't expose it.
- Locates the source PDF via `Dir.glob(Rails.root.join("project/*.pdf")).first`,
  raising a clear error (dev-only, visible to the developer) if absent.

`tmp/manual_pdf_pages/` is already covered by the repo's existing `tmp/`
gitignore — no new ignore rule needed.

### 3. `ManualController` — share `pdfPages`, don't prop-drill

The 87 generated page stubs (e.g. `app/frontend/pages/manual-del-color-vivo/glosario.jsx`)
each destructure only `{ title }` and pass no other props through to
`ManualLayout`. Threading a new `pdfPages` prop through `render inertia:
..., props: {...}` would therefore require editing all 87 stub files to
forward it — real scope creep for a small dev tool, and easy to miss on
future-generated stubs.

Instead, reuse the pattern `InertiaController` already establishes with
`inertia_share user: -> { ... }` (`app/controllers/inertia_controller.rb:4`):
a shared prop is visible to every component via Inertia's `usePage()`,
independent of what the page component itself declares. Add a
controller-level share in `ManualController`, evaluated per-request so it
can depend on `params`:

```ruby
class ManualController < InertiaController
  before_action :authenticate_user!

  inertia_share do
    next unless Rails.env.development?
    range = Manual.pdf_page_range(params[:component])
    { pdfPages: range ? (range.first..range.last).to_a : nil }
  end

  # ...
end
```

Production and test requests never evaluate this block's body meaningfully
(`Rails.env.development?` is false), so `pdfPages` is never part of the
Inertia payload outside development. On the `Index` action `params[:component]`
is nil, `Manual.pdf_page_range(nil)` returns `nil`, so `pdfPages` is `nil`
there too.

### 4. `ManualLayout.jsx`

`ManualLayout` reads the shared prop itself via `usePage().props.pdfPages`
(`import { usePage } from '@inertiajs/react'`) rather than accepting it as a
component prop — this is what makes per-page edits unnecessary, per §3
above.

When `pdfPages` is present and non-empty:

- Render a two-column CSS grid (`grid-cols-[1fr_1fr]` on a wide viewport,
  each column independently scrollable via `overflow-y-auto` with its own
  `max-h-screen`). Left column: a vertically stacked list of
  `<img src={\`/dev/manual_pdf_pages/${n}.png\`}>` for each page number, full
  column width, natural aspect ratio. Right column: the existing
  single-column content (back-link, title, children) exactly as today.
- A small toggle button (top of the left column) collapses it, expanding the
  right column to full width; state is local `useState`, defaults to
  expanded.

When `pdfPages` is absent or `nil` (always true in production/test, and true
in development for the `Index` page and any unmapped path): render exactly
the current single-column markup, unchanged. No visual or behavioral diff
from today.

## Error handling

- Rake task: halts with an explicit, actionable message (title + last good
  page) rather than silently mis-mapping the rest of the book. A human fixes
  the one entry and reruns.
- Runtime: missing PDF file, out-of-range page, or `pdftoppm` failure in the
  dev-only controller surface as normal 404/500s — acceptable since this
  tool only runs on a developer's own machine against their own local PDF
  copy, never in a deployed environment.
- Missing map entry for a given path (shouldn't happen once the rake task
  passes) falls back to single-column, not an error.

## Testing

- Request spec: `GET /dev/manual_pdf_pages/1.png` returns 404 in the `test`
  env (route not drawn — same mechanism the existing Phase 2c spec already
  relies on for env-gated behavior).
- Request spec: `ManualController#show` in `test` env never includes a
  `pdfPages` key in the Inertia page props (asserts current behavior is
  unchanged).
- Unit test for the Levenshtein-fallback helper extracted from the rake task
  (pure function, easy to test in isolation with a couple of known
  near-miss/exact-miss strings).
- No changes needed to the existing 87-route 200-sweep request spec from
  Phase 2c — it doesn't assert on `pdfPages` absence/presence today and
  doesn't need to.
- Manual verification (not automated): `bin/rails manual:map_pdf_pages`
  completes without halting; visiting a section page in `bin/dev`/`serve-dev`
  shows the split view; toggle button collapses it; a production build
  (`RAILS_ENV=production`) shows no split view and no `/dev/manual_pdf_pages`
  route.
